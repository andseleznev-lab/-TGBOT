# PRD: Динамический Zoom-link клуба — уникальные ссылки по дате встречи

**Тикет**: T-013
**Дата создания**: 2026-03-02
**Статус**: PRD_READY

---

## Краткое описание

Клуб проходит **каждую среду в 19:00**. Вместо одной статичной Zoom-ссылки в таблице платежей — создаём нормализованную схему: таблица `club_meetings` (сами встречи) + таблица `club_registrations` (уникальная ссылка на каждого участника). Make.com еженедельно создаёт Zoom-встречу и регистрирует каждого оплатившего участника, выдавая ему персональный `join_url`.

---

## Проблема / Возможность

### Контекст

Участники клуба видят список предстоящих сред. Сейчас:
- `club_payments.zoom_link` — одна ссылка, одинаковая для всех участников и для всех дат
- Zoom-ссылка прописывается вручную
- Нет структуры для хранения расписания встреч отдельно от платежей
- В коде захардкожено «воскресенье», хотя клуб проходит по средам

### AS IS

- `club_payments.zoom_link` — статичная, без привязки к дате
- `getNextSundays()` в боте — расчёт дат по дню недели 0 (воскресенье)
- Make.com не создаёт Zoom-встречи автоматически

### TO BE

- Таблица `club_meetings` — одна запись на каждую встречу (среда)
- Таблица `club_registrations` — одна уникальная Zoom-ссылка на каждого участника каждой встречи
- Make.com: создаёт Zoom meeting → регистрирует каждого участника → сохраняет уникальные ссылки
- Бот: показывает расписание из `club_meetings`, кнопку Zoom — из `club_registrations` пользователя

## Требования

### Функциональные требования

#### Must Have

**БД:**
- [ ] Новая таблица `club_meetings` (`id`, `meeting_date DATE UNIQUE`, `zoom_meeting_id TEXT`, `created_at`)
- [ ] Новая таблица `club_registrations` (`id`, `meeting_id UUID FK`, `telegram_user_id BIGINT`, `zoom_join_url TEXT`, UNIQUE per meeting+user)
- [ ] RLS: `club_meetings` — публичное чтение; `club_registrations` — публичное чтение (фильтрация на клиенте)
- [ ] `club_payments.zoom_link` — больше не используется для бота (оставить как есть, не удалять)

**Make.com:**
- [ ] Новый еженедельный сценарий «Клуб: создать встречу»:
  - Триггер по расписанию (за 2 дня до среды)
  - Zoom API: Create Meeting (среда, 19:00 МСК) → `zoom_meeting_id`
  - Supabase: INSERT в `club_meetings`
  - Supabase: SELECT всех `telegram_user_id` WHERE `status = 'succeeded'` из `club_payments`
  - Для каждого: Zoom API Add Registrant → уникальный `zoom_join_url`
  - Supabase: INSERT в `club_registrations`
- [ ] Изменить существующий payment-сценарий: после подтверждения оплаты — зарегистрировать нового участника в ближайшей предстоящей встрече

**Бот:**
- [ ] Добавить `State.clubMeetings: []` и `State.clubRegistrations: []`
- [ ] `loadClubData()`: загружать `club_meetings` (ближайшие) и `club_registrations` пользователя
- [ ] `renderClubScreen()`: использовать даты из `club_meetings` вместо расчётных воскресений
- [ ] `renderClubMeetingCard()`: `zoomLink` опционален — если нет регистрации → «Ссылка появится позже»
- [ ] Переименовать `getNextSundays()` → `getNextWednesdays()`, день недели: 3 (среда)
- [ ] Обновить все строки «воскресенье» → «среда» в `app.js` и `config.js`

#### Should Have

- [ ] `renderClubScreen()` для экрана до оплаты: показывать предстоящие среды (расчётные, из `getNextWednesdays()`)

#### Won't Have (в этом тикете)

- История прошедших встреч и их ссылок
- Push-уведомление когда ссылка появилась

### Нефункциональные требования (NFR)

**Производительность**: 2 дополнительных Supabase-запроса параллельно при загрузке вкладки «Клуб»

**Безопасность**:
- `zoom_join_url` — не критичный секрет (позволяет войти в Zoom с именем участника), публичное RLS допустимо
- INSERT/UPDATE только через service_role (Make.com)

## Сценарии использования

### Основной сценарий (Happy Path)

1. Make.com (понедельник, 10:00): создаёт Zoom meeting на среду 19:00
2. Make.com: INSERT в `club_meetings` (meeting_date = ближайшая среда, zoom_meeting_id)
3. Make.com: SELECT всех участников (club_payments WHERE succeeded)
4. Make.com: для каждого → Zoom Add Registrant → уникальный `zoom_join_url`
5. Make.com: INSERT в `club_registrations` (meeting_id, telegram_user_id, zoom_join_url)
6. Пользователь открывает бот → вкладка «Клуб»
7. Бот: загружает `club_meetings` + `club_registrations` пользователя
8. Бот рендерит карточки встреч: для встречи со средой текущей недели — кнопка со своим `zoom_join_url`
9. Пользователь нажимает кнопку → попадает в Zoom (все участники — на одну встречу, но по своей ссылке)

### Сценарий 2: Новый участник оплатил после создания встречи

1. Новый участник оплачивает клуб → Make.com payment-сценарий
2. INSERT в `club_payments` (succeeded)
3. Make.com: SELECT ближайшая предстоящая встреча из `club_meetings`
4. Zoom Add Registrant для нового участника → уникальный `zoom_join_url`
5. INSERT в `club_registrations`
6. Участник открывает бот → видит кнопку Zoom для ближайшей среды

### Сценарий 3: Пользователь открывает бот до создания встречи (Make.com ещё не запускался)

- `club_meetings` пустая или нет записи для ближайшей среды
- `club_registrations` для пользователя нет
- Бот: показывает карточки встреч (расчётные среды), кнопок Zoom нет
- Текст: «Ссылка появится позже»

### Edge Cases

**Ошибка Zoom API**: Make.com обрабатывает ошибку; INSERT в club_meetings не происходит; пользователи видят «Ссылка появится позже»

**Дубль INSERT при ретрае Make.com**: `UNIQUE(meeting_date)` в `club_meetings` и `UNIQUE(meeting_id, telegram_user_id)` в `club_registrations` защищают от дублей → `ON CONFLICT DO NOTHING`

**Участник уже зарегистрирован**: Zoom API вернёт существующий `join_url` → Make.com использует его, `club_registrations` upsert
